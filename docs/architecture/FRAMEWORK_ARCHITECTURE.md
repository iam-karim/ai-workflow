# FRAMEWORK_ARCHITECTURE

**Status:** LOCKED
**Version:** 2.0.0

## Purpose
This document establishes the structural organization, major components, and interaction model of the AI-Workflow Engineering OS.

## Architectural Vision
The architecture strictly avoids execution-layer duplication. It acts as the "Governance and Knowledge Brain" that instructs the underlying AI host (Antigravity) on *how* to behave and *what* it is working on, rather than *executing* the work itself.

## Subsystem Justifications
Every subsystem exists because it provides measurable long-term value that the underlying AI host cannot provide natively:
- **CLI:** Because humans need a way to trigger bootstraps and maps manually.
- **Bootstrap:** Because initializing a workspace consistently across CI, IDEs, and AI requires a standardized hook.
- **Workspace/Knowledge:** Because AI lacks global context in massive repos. This solves the hallucination problem.
- **Rules:** Because AI will optimize for short-term fixes without explicit governance.
- **Templates/Prompts/Skills:** Because repetitive engineering- **Core:** Because configuration and ultimate constitutional authority must be grounded.
